# マルチテナント CMS/BaaS 設計仕様書

> 本書は設計議論で確定した事項を固定するための作業ドキュメントである。
> 各決定には根拠と、依存する他の決定への参照を付す。末尾に未処理論点と決定ログを置く。

## 0. 目的とスコープ

理想的な Web システムを自作するための、マルチテナント SaaS。初期段階は一般的なブログ的 CMS だが、内部は kintone 的な「型をデータとして定義する」BaaS 構造を土台に持ち、将来のプラグイン（予約・メール配信等）を無理なく載せられることを狙う。

### 中核の設計原則

- **ローカルファースト最優先**: 編集体験は tanstack-query / tanstack-db によるローカルファースト同期で駆動する。パフォーマンスの主軸。
- **コンテンツ構造の規律 + 実行時定義の自由の両立**: 『Designing Connected Content』の「型付き・連結・再利用可能」という設計規律を、kintone 的な「ユーザー/プラグインが実行時に型を定義する」メカニズムの上で実現する。両者は対立せず、後者のメタモデルを前者の規律で設計する。
- **固定管理画面 + ヘッドレス公開**: 管理画面（バックエンド）は固定 UI を提供する。一方、公開側フロントエンドはヘッドレスで、公開コンテンツ配信 API を介して自由に構築できる（Contentful 型）。
- **Cloudflare 前提**: ホストは Workers、データは Durable Objects (SQLite backend)、バイナリは R2。

## 1. テナント構造（二層）

```
特権テナント（運営者＝自分）
  └─ テナント（利用者） = 1 CMS = 1 Durable Object (SQLite)
```

- **特権テナント**: 全テナントと管理者アカウントの管理を行う。
- **テナント**: CMS を運用し、コンテンツと自テナントに属するユーザーアカウントを管理する。
- **下位区切りは持たない**（当初案の「ワークスペース」層は廃止）。1テナントが複数サイトを必要とする事態が生じたら、その時点でコンテンツ型/関係で表現するか再検討する。

> 注: 当初の三層構造（特権 → 管理者テナント → ワークスペース）および「ユーザー↔ワークスペースの多対多メンバーシップ」は、実課題に照らして不要と判断し廃止した。

## 2. データ分離戦略

### コントロールプレーン / データプレーンの分離

システムは性質の異なる2種類のデータを持ち、これを分離することが構造上の背骨である。

- **コントロールプレーン（ID・認可）**: テナント、ユーザー、メンバーシップ、ロール、課金。認可はワークスペース（テナントのデータ本体）の**外側**に置き、リクエストの入口で必ず検証するゲートとする。「テナントのデータを開いてよいか」を、そのデータを開かずに判定できる必要があるため。
  - メンバーシップは二層化により `(user_id, tenant_id, role)` の素直な形。多対多の交差判定は存在しない。
- **データプレーン（コンテンツ）**: 各テナントの CMS 実体。テナントごとに物理分離する（下記）。

### テナント = 1 Durable Object (SQLite)

- 1テナント = 1つの DO-SQLite。**DO 境界がテナント境界**であるため、コンテンツ系テーブルに `tenant_id` 列は不要で、全クエリからテナント絞り込みが消える。
- 「決して他テナントのデータにアクセスできない」を、論理（全クエリの正しさ）ではなく**物理（データが別 DB に存在しない）**で保証する多層防御。
- DO は各テナントの唯一の書き込み権威（single-writer）であり、ローカルファースト同期のハブとなる。

### 認可フロー

```
クライアント → Worker（認証・メンバーシップ検証） → 該当テナントの DO へルーティング → 同期
```

信頼境界は Worker に固定する。ヘッドレスで API を公開するため、クライアント側の権限申告は一切信用しない。

## 3. インフラとコスト構造

### 採用プリミティブ

- **Workers**: 実行層・認可ゲート・API。
- **Durable Objects (SQLite backend)**: テナントごとのデータ本体 + 同期ハブ。GA 済み、1オブジェクト 10GB。
- **D1**: コントロールプレーン（ID・membership・セッション真実源）＋公開読み取りモデル（テナントごとの publish 投影。第12章）。スケール・トゥ・ゼロ、読み取りレプリカ。
- **Cloudflare Queues**: 非同期副作用フックの搬送（第9章）。
- **R2**: アセットのバイナリ。
- **キャッシュ / KV**: 公開配信の read パス前段、ブロックリスト等。

### コストの要点

DO の課金は「リクエスト / Duration（実行時間）/ ストレージ」の3種だが、実質的に効くのは **Duration** のみ。そして Duration は**ハイバネーション（休眠）できているか**で桁が変わる（公式試算で、同規模でもハイバネなし $138/月 → ハイバネあり $10/月）。CMS はワークスペースが大半の時間アイドルで、ストレージも小さいため、条件を満たせば実質 Workers 有料プラン下限（$5/月）に張り付く。

### コストが強制する設計制約（確定事項）

1. **同期接続は WebSocket Hibernation API 前提**（素の `accept()` は接続時間まるごと課金になるため禁止）。
2. **公開 read パスは DO を経由しない**（キャッシュ / R2 / レプリカに分離。一般閲覧で書き込み権威の DO を起こさない）。

> この2点目が、後述する「非対称二経路」の物理的な根拠になる。

> **コスト攻撃面の注記**: モジュールが公開する無認証 write（第9.7章）は「認証なしで DO を起こせる唯一の穴」であり、ハイバネーション前提を破壊しうる**コスト DoS の攻撃面**である。防御は DO 到達**前**の Worker 層（Turnstile 等の検証＋レート制限）で行い、認可の第1段（第11.5章）と同じ位置で落とす。詳細は第9.7・11.7章。

## 4. メタモデル（型をデータとして持つ）

固定した型でも、生の SQL スキーマをユーザーに開放するのでもなく、**制約付きメタモデル**を採用する。

- 型付きフィールドの**パレットはプラットフォームが固定**する（第5章）。
- **型の定義（＝どのフィールドを組み合わせるか）は、ユーザーもプラグインも実行時に data として作成できる**。

これにより、kintone 的な実行時の自由度、Connected Content 的な構造規律、tanstack-db の型付きコレクション（レジストリから実行時にスキーマ生成）、プラグインの安定契約が同時に成立する。

> プラグインが必要とするのは「拡張性」であって「エンドユーザーによる任意スキーマ定義」ではない（WordPress の実例）。拡張は (1) メタモデルでの型合成、(2) プラグイン専用テーブルという脱出ハッチ、で賄える。

### 4.1 型キーの名前空間

`content_types.key` は UNIQUE だが、**ユーザー型とプラグイン型が同一キー空間を共有すると衝突する**（予約モジュールの `booking` とユーザー作成の `booking` 等）。規約:

- **プラグイン所有型はモジュールIDで名前空間化**する（例: `booking.slot`, `booking.reservation`）。
- ユーザー型は素の名前空間（または `user.` 接頭辞）に置く。
- 登録時に名前空間規約を強制し、越境登録を拒否する。

### 4.2 スキーマ寛容性と型定義マイグレーション

JSON 格納（スキーマレス）である以上、**「古い形の `data` が残り続ける」のが常態**。型定義が変わっても既存 records は自動追従しない。原則:

- **読み取りは寛容（tolerant read）**: 欠損フィールドは既定値/不在として扱い、未知フィールドは無視（ただし破棄せず保持）。records が最新の型定義に適合している保証はない、を前提に読む。
- **書き込みは現行定義で検証（validate-on-write）**: 書き込み時点の型定義でバリデーションし、その record だけが現行形に更新される（遅延適合）。
- **既定で一括変換しない**。フィールドの追加/削除/required化/型変更/renameは、上記の遅延適合で吸収する。rename は「remove + add」の破壊的変更として扱い、値を保持したいなら明示的な移行マッピングを伴わせる。

**プラグイン型の再配布（運用問題）**: `content_types` は各テナント DO 内にあるため、**モジュールのバージョンアップ＝有効化中の全テナント DO への型定義再配布**になる。編成:

- モジュールは**バージョン付きの型定義マニフェスト**を持つ。
- モジュール更新時、コントロールプレーンが有効化テナントごとに**マイグレーションジョブを Queues で配信**し、各テナント DO の `content_types` を更新する。
- マイグレーションは**冪等かつバージョン管理**。データ適合は上記の遅延適合に委ね、DDL を伴う一括変換は避ける。

## 5. フィールド型パレット

### 設計原則: 2層に分ける

- **基底型（storage 層・小さく閉じる）**: SQLite に実際に格納されるもの。
- **フィールド型（authoring 層）**: 基底型 + バリデーション + UI + 意味。ユーザー/プラグインはこちらを見る。

email・URL・slug・通貨・評価スター等は独立した型にせず、基底型 + 制約 + UI ヒントで導出する（モデルと格納・表現の分離）。

### スカラー基底型（この5つに閉じる）

`text` / `number` / `boolean` / `datetime` / `json`

- **通貨は必ず整数の最小単位**で持つ（SQLite REAL の浮動小数誤差を避ける）。
- **datetime は UTC の ISO8601（またはエポック）**で格納し、タイムゾーンは表示層で扱う。
- **json は脱出ハッチ**。中身は関係もクエリも効かない不透明な塊になるため濫用しない。「関係で表せるものを json に押し込む」ことは禁止。

### 構造型

- **enum / select（単一・複数）**: 選択肢は型定義の一部として持つ。
- **リッチテキスト**: **構造化（JSON AST）**を採用（論点A）。本文中のリンクや埋め込みメディアを参照として第一級に持つ。プレーン Markdown ではない。
- **media / asset**: **アセットは独立コンテンツ型**（論点E）。実体は R2 のオブジェクト + メタデータ行。「メディアフィールド」は独立型ではなく `allowedTypes=[asset]` の**関係フィールド**にすぎない。alt / caption は**アセット内在**（文脈依存にはしない）。
- **入れ子コンポーネント / repeater**: **認めない**（論点B、初期段階）。「構造化した小要素の繰り返し」は入れ子ではなく**独立コンテンツ型 + 多対多関係**で表す。

### 関係型（心臓部）

確定した設計:

- **参照であって埋め込みではない**（正規化）。単一の真実源を指す。埋め込みコピーは禁止。
- **カーディナリティは 1対1 / 1対多 / 多対多をサポート**。
- **双方向は片側のみ格納し、逆側は live query の join で合成**（逆リンクを二重保存しない）。
- **「属性を持つ関係」は専用機構を作らず、関係フィールドを持つコンテンツ型として reify する**。例: 予約(booking) = Resource / Customer / 時間枠への関係フィールド + 自身のスカラー（状態・人数）を持つ1コンテンツ型。予約プラグインは新機構を要求せず既存プリミティブの合成で成立する。
- **多態（論点C）**: 指し先を1型に固定せず、**許可された型の集合（allowedTypes）を宣言できる**。プラグインが既存の任意型に接続できる。
- **参照整合性（論点D）**: DB の FK 制約ではなく**ソフト参照**。dangling を許容し、クエリ時に解決する。

### ローカルファーストが強制する帰結（確定）

1. **ID はクライアント生成（ULID/UUID）**。未同期の相手への関係を楽観的に張るために必須。
2. **ソフト参照前提**（上記D）。同期途中の一時的 dangling を正常系として扱う。

### システムフィールド（全型に無償付与）

`id` / `createdAt` / `updatedAt` / `createdBy` / `updatedBy` / `status` / `version`

### フィールド制約は宣言ベース（indexed / unique）

索引や一意性はコードのアドホックな扱いにせず、**型定義の `config` に宣言**する。`"indexed": true`（論点J）に加え、**`"unique": true`** を持てる。

- **一意制約の必要性**: 例えば slug は公開 API の単体取得キー（第12.4章）であり、一意でなければならない。しかし値は `data` JSON 内にあり SQLite の UNIQUE を直接張れない。
- **強制の実装**: (a) DO の書き込み経路で**システム定義 `beforeWrite` フック**として一意性を検証する（第11.5章の認可と同じフック機構に乗る＝一貫性）。(b) 必要なら generated column ＋ UNIQUE index で DB レベルにも張る。
- `indexed` / `unique` はいずれも「制約や最適化を宣言としてデータで持つ」思想で揃い、DO 索引・投影カラム（第12章）・一意検証を一元的に規定する。

## 6. 物理表現（DO-SQLite 内: コンテンツ中核3テーブル + 運用テーブル群）

DO-SQLite にはコンテンツの中核3テーブル（本章）に加え、章をまたいで定義される運用テーブル群が同居する（末尾 6.x に一覧）。まず中核3テーブルを述べる。

物理表現の三択（table-per-type / 汎用 records+JSON / EAV）のうち、**汎用 records + JSON + generated column を基軸**とし、**関係だけ正規化テーブルに分離**するハイブリッドを採用する。

- table-per-type: 実行時 DDL がワークスペースごとにスキーマ版数を分裂させ、メタモデルと二重管理になるため不採用。
- EAV: 1レコード復元に自己 join の山が要り、tanstack-db の typed collection と壊滅的に相性が悪いため却下。

以下は**1テナント DO-SQLite の中身**（`tenant_id` 列が無いのは DO 境界がテナント境界のため）。

### content_types（型定義をデータとして持つ）

```sql
CREATE TABLE content_types (
  id            TEXT PRIMARY KEY,        -- ULID
  key           TEXT NOT NULL UNIQUE,    -- 'article', 'author', 'booking' 等の機械名
  name          TEXT NOT NULL,           -- 表示名
  fields        TEXT NOT NULL,           -- JSON: フィールド定義の配列
  source        TEXT NOT NULL DEFAULT 'user',  -- 'user' | 'plugin' | 'system'
  plugin_id     TEXT,                    -- source='plugin' のとき所有プラグイン
  created_at    TEXT NOT NULL,
  updated_at    TEXT NOT NULL,
  version       INTEGER NOT NULL DEFAULT 1
);
```

`fields` のフィールド定義例（関係フィールド）:

```json
{
  "key": "author",
  "type": "relation",
  "required": true,
  "config": {
    "allowedTypes": ["author"],
    "cardinality": "many",
    "ordered": true,
    "snapshotEmbed": "id"
  }
}
```

- フィールド追加 = この JSON 配列への要素追加 = **DDL ゼロ**。
- `source` / `plugin_id` はプラグイン所有型を同機構で登録するための伏線。

### records（全コンテンツの実体）

```sql
CREATE TABLE records (
  id             TEXT PRIMARY KEY,        -- ULID（クライアント生成）
  type           TEXT NOT NULL,           -- content_types.key を参照（ソフト）
  data           TEXT NOT NULL,           -- JSON: スカラー・enum・リッチテキストAST のみ
  field_versions TEXT NOT NULL DEFAULT '{}',  -- JSON: {fieldKey: counter}。フィールド単位競合検知用（第10章）
  status         TEXT NOT NULL DEFAULT 'draft',  -- ワークフロー専用: draft|in_review|ready|archived（publishedは持たない）
  created_at     TEXT NOT NULL,
  updated_at     TEXT NOT NULL,
  created_by     TEXT NOT NULL,
  updated_by     TEXT NOT NULL,
  version        INTEGER NOT NULL DEFAULT 1  -- record 全体の単調カウンタ
);

CREATE INDEX idx_records_type ON records(type);
CREATE INDEX idx_records_type_status ON records(type, status);
```

**version の規則（第10章の同期と整合）**:

- `version`（record 全体）: DO が**受理したあらゆる書き込みで +1**。用途は publish 陳腐化検知（第7章の `source_version` 比較）＝「公開後に編集されたか」の粗い判定。
- `field_versions`（フィールド単位）: DO が**そのフィールドを書いたときにそのキーの counter を +1**。用途は同期の競合検知（第10章）。関係フィールドも物理的には `relations` に投影されるが、競合検知の対象としてこのマップにキーを持つ（値の格納場所と競合検知は別レイヤー）。
- 両者は DO（single-writer）が書き込み受理時に原子的に更新する。クライアントは各フィールドの最後に同期した `field_versions` を保持し、書き込み時に base version を添えて競合判定に供する。

- system フィールドは実カラム。型固有フィールドのみ `data` JSON に格納。
- **関係は `data` に入れない**（relations テーブルへ）。`data` はスカラー・enum・リッチテキスト AST のみ。
- ホットフィールドは generated column + expression index で昇格。**昇格は宣言ベース**（論点J）: 型定義の `config` に `"indexed": true` を持つフィールドについてのみ、型定義時に一度だけ generated column を張る。自動索引（全 datetime/number）や使用パターン推定は採らない——型が無制限に増える以上、索引の無制限増殖を防ぐため。

```sql
ALTER TABLE records ADD COLUMN g_published_at TEXT
  GENERATED ALWAYS AS (json_extract(data, '$.published_at')) VIRTUAL;
CREATE INDEX idx_records_published_at ON records(g_published_at);
```

- 索引宣言の後付け変更は、型定義の version 更新に伴う軽い DDL として扱う（追加＝generated column 追加で既存行を書き換えない／削除＝drop）。型定義のマイグレーション経路に相乗りし、新機構は不要。

### relations（関係 + 参照インデックスを兼ねる）

```sql
CREATE TABLE relations (
  id            TEXT PRIMARY KEY,        -- ULID
  source_id     TEXT NOT NULL,           -- 参照元 record
  source_field  TEXT NOT NULL,           -- 由来フィールド（本文由来は 'body' 等）
  target_type   TEXT NOT NULL,           -- 多態タプルの型側
  target_id     TEXT NOT NULL,           -- 多態タプルのID側（存在は保証しない＝ソフト）
  ordinal       INTEGER NOT NULL DEFAULT 0,
  origin        TEXT NOT NULL DEFAULT 'field'  -- 'field' | 'body'
);

CREATE INDEX idx_relations_source ON relations(source_id, source_field);
CREATE INDEX idx_relations_target ON relations(target_id);
CREATE INDEX idx_relations_target_type ON relations(target_type, target_id);
```

この1枚が、参照インデックスを別物を作らず兼ねる。効く問い:

- 順引き（この記事の著者は誰か）: `idx_relations_source`
- 逆引き（この著者を参照する記事は）: `idx_relations_target`
- orphan 検出（どこからも参照されないアセット）
- 安全な削除前の使用箇所表示（ソフト参照でカスケードしないため必須）
- 本文中リンクの一元化: リッチテキスト保存時に AST から参照を抽出し `origin='body'` で同じ表に投影

**運用規約**: relations は `data` / AST から導出される**派生データ**。真実源は record 側。record を書くたびに該当 `source_id` + `source_field` の relations 行を消して張り直す（再投影）。DO の single-writer 性により投影更新に競合が起きない。

### 6.x DO-SQLite 内テーブルの全体像

中核3テーブルに加え、各テナント DO には以下が同居する（章をまたいで定義）。この一覧が「1つの DO の中身」の完全な見取り図:

- **中核（本章）**: `content_types` / `records` / `relations`
- **公開版（第7章）**: `published_snapshots`
- **アウトボックス（第12.3章）**: `outbox` — 投影ジョブの原子的記録と排出保証
- **alarm レジストリ（第9.6章）**: `(module_id, next_fire_at)` — 単一 alarm の多重化。sweeper もここに相乗り
- **有効化レジストリ（第9.5・11.5章）**: 有効モジュールID / 適用済みマニフェスト version / モジュール権限展開表

## 7. ドラフト / 公開モデル（論点G）

record 層に「編集中の姿」と「公開されている姿」が同居しうる問題を、**record は1行のまま、公開版だけ別テーブルに退避**する方式で解く。

### published_snapshots

```sql
CREATE TABLE published_snapshots (
  record_id      TEXT PRIMARY KEY,   -- records.id と1対1
  type           TEXT NOT NULL,
  data           TEXT NOT NULL,      -- publish 時点の data のコピー
  relations      TEXT NOT NULL,      -- publish 時点の関係の投影（JSON）
  published_at   TEXT NOT NULL,
  published_by   TEXT NOT NULL,
  source_version INTEGER NOT NULL    -- どの records.version を publish したか
);
```

- **records = 編集の真実源（＝ローカルファースト同期の対象）**。編集者はここを楽観的に書き続ける。
- **publish アクション = 現在の records の姿を published_snapshots にコピーする瞬間**。
- **公開配信 API が読むのは published_snapshots のみ**。編集中の records は決して公開経路に出ない。

### 公開状態の真実源とワークフロー状態の分離（論点M）

「一つの事実は一箇所」の原則により、公開状態とワークフロー状態を別の真実源に分ける。

- **公開状態の真実源 = `published_snapshots` 行の存在**。「公開されているか？」は常に snapshot の有無で答える。`records` 側に公開フラグを持たせない。publish が snapshot を作り、unpublish が消す。
- **`records.status` は編集ワークフロー専用**（`draft` / `in_review` / `ready` / `archived` の固定4値）。`published` という値は持たない。
- この分離により「公開軸」と「ワークフロー軸」が直交し、「公開中かつ編集中ドラフトあり」「未公開ドラフト」「取り下げ済みアーカイブ」等が2軸の組み合わせで自然に表現できる。「status=published なのに snapshot が無い」という不整合状態は、そもそも表現できなくなる。

**「未公開の変更があるか」の導出**（新カラム不要、既存の version 比較で出す）:

- `snapshot なし` → 未公開
- `snapshot あり` かつ `records.version == snapshot.source_version` → 公開済み・未編集（クリーン）
- `snapshot あり` かつ `records.version > snapshot.source_version` → 公開済み・未公開の変更あり（要再 publish）

### 上流決定との整合

- relations は records に対してだけ維持され続ける（F 不変、参照グラフは常に一つ）。公開版の関係は snapshot 内に凍結投影。
- 同期は records、配信は snapshots。経路が物理テーブルで分離。
- records をいくら書き換えても snapshot は publish するまで動かない（編集中も公開版が安定）。

### キャッシュ無効化（論点 I の残件を同時解決）

- **publish がキャッシュ無効化の唯一のトリガー**。
- 通常の編集（drafts の同期）は公開経路に影響せず、キャッシュを触らない。
- publish 時、DO は snapshot を**公開配信レイヤーへ write-through**する（第12章）。これにより一覧・単体クエリの**正確性は投影の更新で自動担保**される。
- 無効化の粒度は**単体 + 一覧の両方**: 当該コンテンツの単体キャッシュに加え、その record が参加する**一覧（型別リスト等）のキャッシュ**も対象。一覧の正確性は投影更新で保たれるため、エッジキャッシュ側は短 TTL（陳腐化を有界化）またはキー単位パージで足りる。
- unpublish は投影行の削除 + キャッシュ削除で対称に扱う。

> 公開 read の物理設計（投影先・クエリ語彙・一覧インデックス）は独立章として第12章に定義する。

### 公開版の関係の展開（論点K・L）: ハイブリッド

- **基本は参照ID のみ焼く (a)**。加えて**不変・準不変の軽い値だけ選択的に埋め込む (b)**。
- 埋め込み可否の線引きは「フィールドの更新頻度と正確性要求」で引き、型定義の関係フィールド `config` に `"snapshotEmbed": "id" | "value"` として宣言的に持たせる。
- **埋め込んでよい (value)**: アセット配信URL・画像寸法・alt テキスト等（実質不変）。
- **参照IDのみ (id)**: 著者の表示名・肩書き・カテゴリ名等（更新の反映が要る）。
- 陳腐化対策は **L1 + L2**: 埋め込むのは不変値だけという規律（L1）+ 古くなったら再 publish で直す（L2）。共有実体変更時の自動再投影（L3）は、実運用で痛みが出てから後付けする。

### archive と公開状態の関係

2軸直交（第「公開状態の真実源」節）ゆえ「`status=archived` かつ snapshot あり（＝公開され続ける）」は**合法だが通常は非意図**。軸を結合させない方針を保ち、**archive 時に unpublish を強制しない**。代わりに:

- archive 操作時に「まだ公開中である」ことを**強く警告**し、unpublish を促す。
- 「archived かつ公開中」の record を**健全性チェックで一覧化**できるようにする。

（強制 unpublish は2軸の直交性を壊すため採らない。**確定**: 警告 + 健全性チェックに留める。）

## 8. 非対称二経路（アーキテクチャ要約）

システムには物理的に分離した2つの経路が存在する。

- **書き込み経路（管理）**: 固定管理画面 → tanstack-db → テナントの DO-SQLite に WebSocket（Hibernation）同期。ローカルファースト。認証済み編集者のみが DO を起こす。真実源は `records`。
- **読み取り経路（公開）**: 第三者ヘッドレスフロント（Astro / Next 等）→ 公開配信 API → **公開配信レイヤー**（DO から publish 時に write-through された派生読み取りモデル。第12章）から配信。**DO を起こさない**。`published_snapshots`（DO 内）は真実源だが公開経路が直接読むことはなく、その投影を読む。

> `published_snapshots` は DO 内にあるため公開経路が直接読むと DO を起こしてしまう。この矛盾は第12章の「publish 時に投影へ write-through、公開経路は投影のみを読む」で解消する。

この非対称は第3章のコスト制約（ハイバネーション前提・公開 read は DO 非経由）と完全に一致する。

### tanstack-db 連携メモ

- コレクションは content_types レジストリから**実行時に生成**し、typed collection として扱う。
- 関係は live query の cross-collection join で解決。多態関係は単一 join で表せず、許可型ごとに解決する（多態のコストとして受容済み）。

## 9. プラグイン（モジュール）システム

### 9.1 正体と信頼モデル

サードパーティ拡張のための汎用プラグイン機構ではなく、**ファーストパーティの機能モジュールをメタモデル上に疎結合に載せる仕組み**とする（論点O）。

- **サンドボックスなし**。信頼できないコードの分離・権限制限・リソース制御（Workers for Platforms 相当）は行わない。「自分専用の理想 CMS」が原則のため不要。
- **外部互換保証・バージョニングなし**。API 境界を第三者に対して永久保証する負担を負わない。
- モジュール境界の価値は「他人に書かせる」ことではなく、**リーンなコアの実現**にある。有効化していないモジュールのコードは走らない（当初の不満「多機能ゆえのパフォーマンス低下」への直接の解）。

### 9.2 モジュールが触れる4面

- **面1 データ拡張**: `content_types` に型を登録（`source='plugin'` / `plugin_id`）。既存機構そのまま、新規設計不要。
- **面2 サーバーロジック（フック）**: コンテンツのライフサイクルに反応。下記2種。
- **面3 管理画面 UI 拡張**: スロット方式を基本、複雑な画面のみ独自ルート（論点P）。
- **面4 外部インターフェース**: 公開エンドポイント / 外部送信 / Webhook。Queues 経路に乗る。

### 9.3 フック機構（論点N: ハイブリッド）

- **同期バリデーションフック（DO 内）**: 書き込み経路内で実行し、書き込みを**拒否できる**。single-writer なので検証と書き込みが競合なく原子的。例: `booking.beforeWrite` で空き枠検証。
- **非同期副作用フック（Cloudflare Queues）**: DO はコミット後にイベントを発火するだけで即ハイバネート。consumer Worker が外部副作用（メール送信・Webhook）を DO の外で処理。
- **確定制約**: Queues は at-least-once 配信のため、**副作用フックは冪等必須**（例: `send_log` 照会で二重送信防止）。

### 9.4 書き込み経路へのフック挿入

```
1. DO が書き込みを受ける（single-writer）
2. 有効モジュールの同期 beforeWrite フックを順に実行 → 拒否されれば中断
3. records に書き込み + relations を再投影
4. コミット
5. afterWrite / afterPublish イベントを Queues に発火 → 即ハイバネート
6. consumer Worker が拾い、冪等に副作用を実行
```

ステップ2が DO 内なのは整合性（single-writer の強み）、ステップ5-6が外なのはコスト（ハイバネーション）。第3章と整合。

### 9.5 モジュール有効化（論点Q: テナントごとレジストリ）

- 各テナントの DO 内に**有効モジュールIDの config** を持つ（DO 自身がフック配送のため自分の有効モジュールを知る必要がある）。
- 無効なモジュールのフックは配送されず、コードが走らない。ブログ用途のテナントは予約を無効化＝予約コードは一切実行されない。
- ビルド時静的（全テナント同一機能セット）は、使わない機能を全テナントに積むことになり不採用。
- **適用済みマニフェスト version の記録**: この有効化レジストリに、モジュールごとの「適用済み型定義マニフェスト version」（`(module_id, applied_manifest_version)`）も同居させる。第4.2章のマイグレーション配信が冪等・バージョン管理であるための、各 DO 側の適用状態の置き場。第11.5章のモジュール権限展開表もここに同居する。

### 9.6 時限実行（論点R: DO alarm の疎な運用）

- メール予約送信・予約リマインダー等の時刻トリガーは **DO alarm** で実現。
- **重要な制約**: Durable Objects の alarm は **1オブジェクトにつき1本**。複数モジュールがそれぞれ alarm を持つことはできない。したがって **DO コアが単一 alarm を所有する多重化レイヤー**を持つ:
  - 各モジュールは「次回処理時刻」をコアの**alarm レジストリ**（DO 内の小テーブル/config: `(module_id, next_fire_at)`）に登録する。
  - コアは全登録の**最小時刻で `setAlarm`** する。
  - 起床時、コアは期限が来たモジュールへディスパッチし、処理後に次の最小時刻で**張り直す**。
- record ごとに時刻を持たない設計は維持（モジュール単位の集約で DO の起床を最小化）。

### 9.7 公開経路からの write（非対称二経路の例外）

- 予約の「公開サイトから予約作成」は、第8章の「公開経路は read only」に対する**明示的な例外**。認証済み編集者以外が DO を起こしうる唯一のケース。
- 規約: **モジュールが明示的に公開した write エンドポイントだけ**が DO に到達しうる。濫用防止（レート制限・検証）は当該エンドポイント定義時の責務。

### 9.8 当初動機での検証結果

- **予約**: `resource` / `time_slot` / `booking` を型登録。`booking` は関係フィールド＋状態スカラー（「属性を持つ関係のコンテンツ型化」の実例）。空き検証は DO 内同期フック、確認メールは Queues 非同期。→ 新機構ゼロ、既存プリミティブの合成で成立。
- **メール配信**: `campaign` / `subscriber` / `send_log` を型登録。バッチ送信は Queues（send_log で冪等）、予約送信は DO alarm、外部送信は面4の egress。

### 9.9 面3 UI の実装方針（論点P）

- スロット方式を基本とし、管理画面コアが拡張スロットの語彙（ナビ項目・編集画面パネル・ツールバー等）を定義する資産を持つ。
- 複雑な画面（予約カレンダー等）のみ独自ルート `/modules/{id}/*` に逃がす。
- スロットの具体的語彙リストは UI 実装フェーズの仕事。本仕様では「スロット方式を採る」方針の確定のみ。

## 10. 同期プロトコル

### 10.1 前提（既存決定による制約）

同期設計の大枠は既存決定でほぼ確定している。

- **同期単位 = テナントの DO**（single-writer の権威）。クライアントは楽観的に書き、DO が真実源。
- **同期対象は `records` のみ**。`published_snapshots` は公開経路の別物で非対象。`relations` は派生データで非対象（DO 側で record から再投影）。
- **輸送は WebSocket（Hibernation API）**。プッシュ型。
- **ID クライアント生成（ULID）＋ソフト参照**。
- クライアントは tanstack-db の型付き正規化コレクションで `records` を保持。

大枠は「WebSocket 上で record 単位の変更を、DO を真実源として双方向に流す」に固まる。

### 10.2 差分の粒度（問い1）

- **record 単位で転送**。CMS の record は小さく（数KB）、同時編集頻度も低いため、フィールド単位パッチの複雑さは便益に見合わない。
- 例外余地: 構造化リッチテキスト本文が肥大化して問題になった場合のみ、本文フィールドを別扱いする余地を残す。

### 10.3 競合解決（問い2・3）

single-writer の DO が競合を裁定する。分散マージの難問が「DO が順に受けて裁く」に縮む。基本は**フィールド単位マージ**:

- **別フィールドの変更** → 両立（マージ）
- **同一スカラーフィールドの競合** → LWW（後勝ち。スカラーは小さく実害小）
- **同一本文フィールド（AST）の競合** → **検知して手動解決（論点S = c）**

競合検知の基盤は `records.field_versions`（第6章）。DO はフィールドごとの counter を書き込み時に進め、クライアントが添える base version と比較して競合を判定する。

**関係フィールドの扱い（精密化）**: 同期対象外なのは `relations` の**逆引き投影・派生インデックス**であり、関係フィールドの**前向きの値**は record の一フィールドとして同期される（`field_versions` にキーを持つ）。関係フィールドの同一フィールド競合は既定で LWW（フィールド値まるごと後勝ち）。集合的マージ（要素の add/remove を保つ）は将来オプション。いずれにせよ最も厄介な「逆引きグラフの分散マージ」は、record からのサーバー側再投影で回避される（第6章）。

**CRDT は不採用**。固定管理画面・低同時編集頻度という用途に対し、CRDT のデータ肥大と実装コストは過剰。将来「同時本文編集が頻繁で手動解決が苦痛」という痛みが出た場合に限り、本文フィールドに限定した CRDT/OT をオプションとして足す。

### 10.4 本文競合の検知機構（論点S = c の含意）

新規機構は不要。競合解決の基盤である**フィールド単位 version** により、DO は「送られてきた変更のベース version が現在より古い」ことを検知できる。本文フィールドでこれを検知したら自動マージせず、クライアントに競合を返してユーザーに採用選択を促す。フィールド単位 version の自然な延長。

### 10.5 オフライン再接続

競合解決と同じ機構で吸収する。

- オフライン中の変更は「フィールド単位 version 付き変更の列」として溜まり、再接続時に DO へ送られ、10.3 のルールで順に裁定される。
- ULID とソフト参照により、オフライン中に作成した新 record とその関係も、ID 衝突なく・dangling を許容して送れる。新機構不要。

## 11. 認証・認可

### 11.1 認証と認可の分離

- **認証（AuthN）**: 「誰か」を確立する（ログイン・セッション・トークン）。
- **認可（AuthZ）**: 「何をしてよいか」を判定する。構造は第2章で決定済み（membership を入口ゲートで検証、信頼境界は Worker）。

### 11.2 認証方式とコントロールプレーンの置き場（論点T）

**中央 D1 ＋ 短命 JWT のハイブリッド**。

- ユーザー・membership・セッションの真実源は**中央 D1（コントロールプレーン）**。
- ログイン時に**短命 JWT（例: 15分）**を発行し、身分情報を載せる。有効な間、Worker は署名検証だけでルーティングでき、D1 を引かない（速い・安い）。
- 失効: 短命ゆえ最大15分で自然失効。加えて重大操作（権限剥奪・BAN）は**ブロックリスト照会**で即時に効かせる。ブロックリストは**KV（または Worker キャッシュ）に置き、更新は稀**（失効イベント時のみ書く）。毎リクエスト D1 を引かないため「D1 を引かない」利点を損なわない。
- 認証のたびに DO/DB を無駄に起こさない（第3章のコスト思想と一致）。

### 11.3 ロールモデル（論点U）

**RBAC を基本**とする。

- デフォルトロール `owner` / `editor` / `viewer` を用意し、大半のユーザーはこれで済む。
- 権限は**「型 × 操作」の粒度**で表現（動的に増える型に対応）。
- モジュールが**独自権限を宣言できる**（例: 予約モジュールが `booking:approve`）。モジュールシステム（第9章）と同じ「宣言をデータで持つ」思想。

### 11.4 JWT に載せるもの / 載せないもの（T × U の含意）

権限セットが動的・大規模になりうる（型が増えるほど権限が増える）ため、JWT には**変わりにくく小さいものだけ**載せる。

- **載せる**: `user_id` / `tenant_id` / `role`（粗い役割名）/ 発行・失効時刻。
- **載せない**: 型 × 操作の権限マトリクス。`role` 名からサーバー側（Worker/DO）で展開する（role → 権限の展開表は中央 D1 またはモジュール定義にある）。
- JWT は「粗い身分証」に留める。細かい認可はサーバー側の権限表で行うため、型が増えても JWT は陳腐化しない。

### 11.5 認可の2段階（2場所）

型 × 操作粒度を入れた結果、認可は1箇所では終わらず、情報のある場所に応じて2段に分かれる。

- **第1段（Worker 入口・粗い）**: 「このユーザーはこのテナントのメンバーか」。JWT 署名検証 ＋ `tenant_id` 照合のみ。通らなければ DO を起こさない（コスト保護＝第3章）。
- **第2段（DO 内・細かい）**: 「*この型のこの操作* をしてよいか」。型 × 操作の権限表を見る。`content_types` とモジュール権限宣言が DO 内にあるため、ここでしか判定できない。
- 第2段は書き込み経路の `beforeWrite` フック（第9章）と**同じ場所・同じ機構**で走る。認可チェックは実質「システム定義の `beforeWrite` フック」として、モジュールのフックと同じ仕組みに乗る。
- **権限展開表を DO へ届ける経路**: 判定は DO 内で走るため、展開表が DO ローカルに要る。**デフォルトロール（owner/editor/viewer）の権限はコードに焼く**（アプリと共にデプロイ）。**モジュール宣言の権限は、モジュール有効化時にテナント DO の config へ書き込む**（第9.5章の有効化レジストリと同じ経路）。これで第2段判定は DO 内で完結し、外部照会を伴わない。

### 11.6 特権テナントの認証（論点V）

- **別ログイン経路**とし、可能なら多要素。権限の非対称が極端なため、通常ユーザーと同じ扉に置かない。
- 「全テナント管理」操作は通常の第2段認可を飛び越える **super 権限**。その行使は**監査ログに残す**（強い権限には記録を伴わせる）。

### 11.7 公開経路の write（論点W）

- 認証系から切り出し、「**公開エンドポイントの責務**」として濫用防止層で守る。
- 公開 write は**第1段（メンバーシップゲート）を持たない**経路。代わりに濫用防止層（レート制限・入力検証・エンドポイントごと許可リスト）を第1段の位置に置く。
- 第2段は残る: モジュールが「この公開エンドポイントは `booking:create` のみ許可」と宣言する。
- 詳細仕様は実装フェーズ送り。

## 12. 公開配信レイヤー（公開 read の物理設計）

第8章の「公開 read は DO を経由しない」と第7章の「公開が読むのは `published_snapshots`」は、`published_snapshots` が DO 内にある以上そのままでは両立しない。この章がその矛盾を解く。原則は **CQRS 的な読み取りモデルの分離**——真実源（DO 内 snapshot）から、公開専用の**派生読み取りモデル**へ publish 時に write-through する。

### 12.1 真実源と読み取りモデルの分離

- **真実源**: `published_snapshots`（DO 内）。publish の凍結像。公開経路は**ここを直接読まない**（読むと DO が起きる）。
- **公開読み取りモデル（publish 投影）**: DO の外にある**クエリ可能な派生ストア**。公開 API はここだけを読む。DO を起こさない。

### 12.2 投影先: 共有投影 D1 ＋ `tenant_id`（プラットフォーム制約による決定）

**プラットフォーム事実**: D1 の Worker バインディングは wrangler に**静的宣言**が必要で、DO の `idFromName()` のような実行時アドレッシングが存在しない。テナントごとに D1 を動的にバインドする手段は現状なく、唯一の方法は Workers API での再デプロイ（Cloudflare 公式回答、ETA 未定）。REST API 経由はアカウント全体のレートリミットを共有するため、最もトラフィックの多い公開 read 本線には載せられない。よって **per-tenant 投影 D1 は現行プラットフォームで実装手段が事実上ない**。

したがって投影先は **`tenant_id` 列を持つ共有 D1**（静的にバインド可能）を採る。

**脅威モデルの再整理（なぜ投影は共有でよいか）**: 書き込み側で物理分離を選んだ論拠は「非公開データの秘匿性を、論理でなく物理で守る」だった。しかし**投影に入るのは publish 済み＝公開を意図したデータのみ**であり、守るべき秘匿性が原理的に薄い。分離の論拠は投影には外挿できない。行レベル分離の残余リスク（クエリの絞り込みミス）で漏れるのは「いずれ公開される/された情報」であり、実害が桁違いに小さい。ゆえに投影側だけ「コンテンツ系に `tenant_id` を持たない」原則（第2・6章）を破ることは、**「投影は真実源ではない使い捨ての派生物」**という位置づけで整合的に説明できる。

**成長路線**: 単一共有 D1 は 1DB 10GB 上限とクエリ集中がいずれ効く。規模が育ったら**固定 N 本のシャード D1**（静的バインド可能な本数）に tenant をハッシュ配置する。共有案の自然な発展形で、10GB 上限とロード分散の双方に効く。per-tenant の完全分離が本当に要るなら Workers for Platforms での User Worker 個別バインドは可能だが、9.1 で WfP 相当を不採用としており投影のためだけの導入は過剰。

**投影の物理設計（3テーブル）**: 共有 D1 では全テナント・全型が一枚の records 投影を共有するため、**per-tenant 時代の「宣言フィールドを実カラムに昇格」戦略は成立しない**（(テナント × 型) ごとに異なる昇格カラムが一枚に載らず、カラムが際限なく増え名前も衝突する）。代わりに2段構えで索引を効かせる。

- **records 投影**: `(tenant_id, record_id, type, slug, published_at, data, source_version)`。**システム共通フィールドのみ実カラム**（どのテナントでも意味が同じもの）。複合索引 `(tenant_id, type, published_at)`・`(tenant_id, type, slug)` を張り、**一覧クエリの大半（最新N件・slug 単体取得）はこれで完結**。型固有フィールドは `data` JSON に残す（実カラムに昇格しない）。
- **relations 投影**: `(tenant_id, source_id, source_field, target_type, target_id, ordinal)`、**published なもののみ**。`(tenant_id, target_id, source_type)` 等に索引を張り、メンバーシップフィルタ（「カテゴリXの記事」）と逆引きを O(log n) で解く。
- **projection_index（索引専用サイドテーブル）**: ユーザー宣言の `indexed` フィールドはここへ。`(tenant_id, type, field_key, value_text, value_num, value_date, record_id)` で `(tenant_id, type, field_key, value_*)` に索引。フィルタ/ソートでここが `record_id` を絞り、実体は records 投影の `data` へ**1回 join** で取る。

> **EAV との区別（第6章の却下理由と混同しないための一文）**: projection_index は形の上では EAV だが、第6章が EAV を退けた理由（「1レコード復元に自己 join の山」）は**当てはまらない**。この表は**レコードを復元するためではなく、フィルタ/ソートで `record_id` を絞るためだけ**に使い、実体は records 投影から1回の join で取る。relations 投影が「メンバーシップフィルタ専用の索引」であるのと同じ役割分担で、既にある投影の思想と揃う。**復元用 EAV は却下・索引用サイドテーブルは可**。

- **値の型付き格納（正しいソート/範囲のため）**: projection_index の値は型別カラム（`value_text` / `value_num` / `value_date`）に振り分ける。数値・日付を TEXT 一列で持つとソート/範囲比較が壊れる（数値の文字列ソート問題）。フィールドの型は `content_types` から既知なので、クエリ層が適切な列を選べる。
- **複数 indexed フィールドの同時フィルタ**は索引2引きの交差になるが、公開一覧の複雑度として許容範囲。12.4 の「フィルタ/ソートは索引宣言済みフィールドに限る」という語彙制限が、ここでも複雑度の上限として効く。

**ランタイム DDL の受容（範囲が縮小）**: 共有 D1 化により、宣言フィールドは実カラムでなく projection_index の行になったため、**型定義のたびに投影へ DDL を打つ必要はほぼ消えた**（records 投影・relations 投影・projection_index の3スキーマは固定）。残る DDL はシステム共通カラムの変更程度で稀。いずれにせよ投影は snapshot から再構築可能な派生物であり、版数分裂は再投影で解消できる（第6章の table-per-type 却下と衝突しない理由は同じ）。

### 12.3 投影の更新: アウトボックス ＋ 非同期冪等投影

**dual-write 問題**: DO-SQLite のコミットと投影 D1 への書き込みは**別システムへの2書き込みで、分散トランザクションがない**。同期 write-through にすると D1 側失敗（障害・レート制限）で真実源と投影が乖離し、「snapshot はあるのに公開されない・編集者には成功に見える」という故障モードを生む。したがって同期 write-through は採らず、**アウトボックスパターンで Queues に相乗り**させる。

- **publish/unpublish 処理**: DO-SQLite 内で snapshot 作成/削除 ＋「投影ジョブ」をアウトボックスとして記録し**コミット**（ここまで単一 DB 内で原子的）。
- **投影の upsert/delete**: Queues 経由の consumer が**冪等に**実行（第9.3章「副作用フックは冪等必須」と同じ規約に乗る）。
- **順序逆転への防御（upsert）**: 投影行に `source_version` を持たせ、「**現在の投影より古い version の upsert は無視する**」ガードを入れる（publish 直後の unpublish が先に着く等に耐える）。
- **順序逆転への防御（delete）**: delete ジョブにも**発行時点の version を載せ**、「**投影の `source_version` がそれより新しければ delete を無視**」する。unpublish 発行後に republish が先着し、遅れて届いた delete が消してしまう事故を防ぐ。
- **アウトボックスの排出保証（sweeper・クラッシュ耐性）**: 拾い直しを **sweeper** が担い、**9.6 の alarm 多重化レジストリにシステムモジュールとして相乗り**する。ただし「enqueue 失敗を観測したら登録」では、コミット直後・enqueue 前に DO が落ちた場合（evict・デプロイ・例外）に登録者が不在となり publish が宙吊りになる（低トラフィックなテナントほど長引く）。したがって保証を「失敗の観測」ではなく「**コミットの事実**」に依存させる: **outbox 行を書くのと同一コミットで sweeper の `next_fire_at`（数秒後）を alarm レジストリに登録**する。両者は同じ DO-SQLite 内なので単一トランザクションで原子化できる。sweeper は起床時に未送出行を再送し、**全行送出済みなら no-op で登録を消す**（冪等ゆえ正常系で毎回登録しても実害なし）。
- **投影 consumer の3テーブル更新の原子性**: republish の upsert は records 投影の行更新に加え、その record の relations 投影・projection_index の**旧行削除→再挿入**（DO 内 relations 再投影と同じパターン）を伴う。version ガードの判定（record 単位で一度）と3テーブルの置き換えは D1 の `batch()`（暗黙トランザクション）で一括実行し、半端な状態を残さない。
- 代償は「publish が投影に反映されるまで数秒の結果整合」。公開配信では完全に許容範囲。

編集（draft 同期）は投影を触らない。第7章「publish がキャッシュ無効化の唯一トリガー」とも整合（無効化もこの consumer が担える）。

> **アウトボックステーブル**（DO 内）: `outbox (id, job_type, record_id, source_version, enqueued_at, sent)`。`job_type` は upsert/delete、`sent` は送出済みフラグ。スキーマは第6章の運用テーブル群に置く。**掃除**: `sent=1` の行は単調増加するため、送出確認後に削除、または一定期間後に purge する（デバッグ用に短期保持は可、方針だけ固定）。

### 12.3b 投影は再構築可能な派生物（再投影ジョブ）

**原則**: 投影はいつでも DO 内 snapshot から**決定的に再構築できる使い捨ての派生物**である。これは `relations` が record から再投影で復元可能な派生データ（第6章）であることの完全な相似形で、設計思想の統一になる。

テナント単位の**再投影ジョブ（Queues 経由・冪等）**を第一級の機構として持つ。用途:

- dual-write 乖離の最終防衛線（乖離検知時に snapshot から作り直す）。
- **`indexed` の後付け変更のバックフィル**: 新たに `indexed` 宣言したフィールドは、既存の公開済み record について projection_index の行が無い。再投影が records 投影の `data` から値を抽出して projection_index を埋め直す（DO 側の VIRTUAL generated column は既存行を書き換えないが、投影側のサイドテーブルはバックフィルが要る、という非対称）。
- 型定義変更・投影スキーマ変更・障害復旧・（将来）シャード再配置。

> **共有 D1 への書き込み集中（運用注記）**: 通常の publish 頻度では問題にならないが、**テナント全体の再投影ジョブ**は共有 D1 の書き込みプライマリに負荷が集中しうる唯一の経路。再投影は**バッチ化し、consumer の並列度を絞る**こと。

### 12.4 公開 API のクエリ語彙

投影 D1 に対して、公開 API は最小限の語彙を提供する:

- **単体取得**: id または slug で1件。
- **一覧取得**: 型指定 + フィルタ（**索引宣言済みフィールド**の等値・関係メンバーシップ）+ ソート（索引宣言済みフィールド、例 `published_at` 降順）+ ページネーション（カーソル/limit）。
- フィルタ/ソートは**索引宣言済みフィールドに限る**（無制限クエリは許さない＝投影のカラム設計と一致）。
- **複数値フィールドの意味論**: enum（複数選択）を indexed 宣言すると projection_index は「1値=1行」で複数行になり、フィルタは自然に any-of 意味論になる（row-per-value の利点としてそのまま受容）。ただし**ソート可能なのは単一値の索引フィールドのみ**（複数値フィールドのソートは未定義）。

### 12.5 公開経路での関係解決

- 公開経路の関係解決は**投影（published のみ）に対してのみ**行う。参照先が未 publish / unpublish 済みなら、その参照は**不在として扱う**（ソフト参照の公開版）。
- `snapshotEmbed=value` の埋め込み値は投影行にインラインで載る（解決不要）。`snapshotEmbed=id` は投影内で解決し、解決先が非公開なら不在。
- **編集 UX**: 未公開の参照先を持つ record を publish しようとしたら**警告**する（公開すると当該リンクが不在になる旨）。ソフト参照方針によりブロックはしない。

### 12.6 キャッシュ戦略

- 一覧・単体の**正確性は投影更新で担保**される（publish から数秒の結果整合。コールドキャッシュ時は投影 D1 を読む＝DO 非経由、404 にならない）。
- エッジキャッシュ（Cache API / KV）は投影の**前段の遅延最適化**。無効化は publish/unpublish 時にキー単位パージ、または短 TTL で陳腐化を有界化。一覧の厳密なタグ無効化に依存しなくても、投影が常に正のため実害が小さい。

### 12.7 コスト整合

- 公開 read は投影 D1（スケール・トゥ・ゼロ、レプリカ）とエッジキャッシュのみを叩き、**DO を起こさない**——第3章のハイバネーション前提を維持。
- 唯一の例外は第9.7章の公開 write（DO 到達）で、これは第3章のコスト攻撃面注記のとおり Worker 層で DO 到達前に防御する。

## 13. 非目標（Non-Goals）

以下は CMS として必ず問われるが、**現時点では意図的にやらない**。「考慮した上での非目標」であり「見落とし」ではない。各項目に、目標化する際に**再訪すべき既存決定**を添える（将来の格上げ時の地図として）。

- **全文検索**: 現状スコープ外。筋道は明確で、**投影 D1（第12章）に SQLite FTS5 で全文索引を足す**のが既存構造に最も素直に乗る（公開検索の置き場＝投影 D1）。目標化時に第12章の投影テーブル設計に FTS を追加する。
- **i18n / 多言語**: スコープ外。**最も緊張が強い**——alt/caption を**アセット内在**にした決定（第5章・論点E）は多言語と構造的に衝突する（多言語では alt は「アセット固有」でなく「アセット×言語」になる）。目標化するなら、alt 内在の判断と、より広く「コンテンツの言語軸をどうモデル化するか（型ごとロケール別 record か、フィールド内ロケールマップか）」を再訪する必要がある。第5回で「文脈依存 alt は入れ子禁止(B)と衝突するため内在にする」とした判断が、ここで再浮上する。
- **アセット変換パイプライン**: スコープ外。R2 にバイナリを置く（論点E）までは決定済みだが、画像リサイズ・variants 生成は未定義。**Cloudflare Images に逃がす**か Workers で変換するかは、公開 read パス（DO 非経由・第12章）との相性を見て決める。目標化時は配信経路への影響を要検討。
- **バックアップ / テナントのエクスポート**: スコープ外。DO-SQLite の PITR（Time Travel）は「時間を巻き戻す」であって「テナント単位の**可搬な形式で吐き出す**」とは別問題。メタモデルゆえ、エクスポートは「型定義（content_types）＋records＋relations＋アセット」を一括で可搬にする**形式設計**が要る。真実源が DO 内にあるため、DO からのダンプ経路も設計対象。
- **DO のロケーション（地理分散編集）**: **前提として明記**。DO は単一リージョンに固定されるため、編集者が地理的に分散すると同期レイテンシに直撃する。本設計が「自分専用の理想 CMS」（第14決定＝原文の設計前提）である限り**非問題**。将来テナントを他者に開き、かつ編集者が広域分散する運用を目標化するなら、この前提が崩れ、DO ロケーション戦略（location hint、リージョン選択、あるいは編集側の再設計）が必要になる。
